12
Pressure Drop — PFR
🔑 Momentum balance adds a 3rd ODE to the PFR system — architecture decision needed
Add optional Ergun equation to the PFR ODE system: dP/dz = -G/ρ·(1-φ)/φ³ · (150(1-φ)μ/Dp² + 1.75G/Dp). New PFR node toggle "Pressure Drop" with inputs: catalyst particle size Dp, void fraction φ, inlet pressure P0. The ODE becomes [Ca, Cr, Cs, T, P] (5-vector). The rate law is multiplied by (P/P0)ⁿ for gas-phase. Architecture: pfrModel already integrates a vector ODE — extend the state vector from [Ca,Cr,Cs,T] to [Ca,Cr,Cs,T,P] when pressure-drop mode is enabled. New pure file: pressureDropModel.ts.
Enables
constant-pressure PFR only → packed bed PFR · Ergun ΔP · optimal bed length design

Gas-Phase Reactions

🔑 Architectural decisions to get right upfront — 2 new from Graphify analysis
reactor.ts type split: math types vs. UI types
✗ Keep all types in one file — "they're just types, imports are cheap." This approach lets reactor.ts continue bridging 9 communities; a single field rename cascades across the entire project.
✓ Extract KineticsType, ReactorType, ReactionMode, UnitType, ThermalMode → src/types/simulation.ts. Pure math files only import from simulation.ts. UI node-data shapes stay in reactor.ts. Bridge count drops 9 → ~4. Compiler enforces the boundary — if a pure file imports from reactor.ts, purity test catches it.

Graphify confirms this is the single highest-risk file. The split is zero-runtime-cost and the compiler does all the verification.

SimulationParams: stable interface with migration checklist
✗ Add fields to SimulationParams freely as P4 features need them — it's just adding optional fields, how bad can it be? (Answer: 19 dependents, 7 communities, breaks serializer migration without an explicit bump.)
✓ Treat SimulationParams as a versioned interface. Before adding any field: (1) grep for all spreads ({ ...params }) and destructures, (2) add the field + default to the serializer migration map in serializer.ts, (3) check sweepEngine, targetSolver, comparisonEngine — they all take params, (4) run full purity test suite. This checklist is in the JSDoc comment on the interface definition.

P4.2 adds gasPhase + epsilon, P4.4 adds Q_feed — two migrations. Each is manageable if done carefully, risky if rushed.

Splitter stream assignment: handle-name vs. edge-order
✗ Continue matching sourceHandle === 'out-top'/'out-bot' — breaks when user connects via any other handle
✓ Fall back to edge-index: outEdges[0] gets α fraction, outEdges[1] gets (1−α). Keep handle-name check as the primary path, add index fallback. Guard stale edges (source/target not in nodeMap) before topoSort. Done in P1.1.

The splitter has always had two distinct source handles for a reason (top/bot fraction assignment). But the index fallback makes it robust to ReactFlow handle connection behaviour variance.

Per-node validation: pure vs. context
✗ Import validation results inside each node component — breaks purity test for validation.ts
✓ validation.ts stays pure (returns nodeIssues: Map<id, issue[]>). A new ValidationContext (React context, not Zustand) distributes the map to nodes. Nodes read from context, not store.

Validation context is the right seam: topology-local information flows naturally as React context, not global Zustand state.

PFR ODE state vector extension for pressure drop
✗ Add a separate pressure-drop ODE pass after the main integration — doubles the integration cost and makes T/P coupling incorrect
✓ Extend the RK4 state vector from [Ca, Cr, Cs, T] to [Ca, Cr, Cs, T, P] when pressure-drop is enabled. The existing rk4Step already handles arbitrary-length vectors. Add a pressureDrop flag to UnitParams.

T and P are coupled (temperature affects viscosity, which affects ΔP). They must be integrated together in one RK4 step.

Dark mode strategy: Tailwind class vs. CSS variables
✗ Use prefers-color-scheme media query in CSS only — can't be overridden by the user's manual toggle
✓ Tailwind darkMode: 'class' strategy. A themeSlice in Zustand holds 'light' | 'dark' | 'system'. On mount, check matchMedia; on toggle, flip the dark class on <html>. Persisted in localStorage.

Class-based dark mode is the only approach that supports both system-preference detection AND user override with a single toggle.

Phase 3 purity rule: The pure-function invariant is preserved. New pure files added in P4 (pressureDropModel.ts, gasPhaseFactor.ts, steadyStateMapper.ts) get added to PURE_FILES in purity.test.ts. src/types/simulation.ts (from P1.4) is a type-only file — add it to PURE_FILES too. New src/schema/parameterSchema.ts (from P3.3) is also pure. The validation map return from validation.ts is type-only; the ValidationContext React context lives in a separate src/context/ file. Tests: 17 green now → target 28+ tests after all Phase 3 pure files (17 + ~3 type-split tests + ~4 P4 math file tests + 4 for new pure schema/context files).  Graphify invariant: keep zero import cycles. Every new file must be verified cycle-free before merge. SweepPanel.tsx (21 outgoing deps) is a known hotspot — do not add new imports to it in P3.3; move definitions out instead.

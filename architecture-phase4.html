Cuts Reactor Nodes ↔ Param Panel from 33 edges → target ≤ 15 · eliminates 3× duplicated pattern flagged by Graphify

13
Semibatch Reactor
Implements the "CA high, CB low" strategy from Levenspiel Fig. 7.1
Semibatch (or fed-batch) reactor: a vessel that starts with species A, and B is fed continuously at rate FB₀ (mol/s). This keeps C_B low throughout, maximizing selectivity S_R for reactions where the desired product favors low C_B (e.g., A→R if rR depends on C_A only; B→S competes if C_B is high). ODE system on a moles basis: dNA/dt = rA(CA,CB)·V, dNB/dt = FB₀ − rB·V, dNR/dt = rR·V, dV/dt = FB₀/CB_feed (volumetric addition rate). Concentrations: CA = NA/V, CB = NB/V. Solved with RK4 from t=0 to t=tau_batch. New node params (in node.data): tau_batch (total time, s), FB0 (B feed rate, mol/s), CB_feed (B feed concentration, mol/L). Result: final NA, NB, NR → Xa = (NA₀ − NA)/NA₀, S_R = NR/(NA₀ − NA). New pure file: src/math/semibatchModel.ts. New node type: semibatch (icon: batch vessel with side-entry nozzle on the right).
Enables
batch = fixed-charge, no selectivity control → fed-batch · CB controlled low · selectivity optimized for parallel rxns

14
Side-Injection PFR
Implements the "CA high, CB distributed" pattern from Levenspiel Fig. 7.2
A PFR variant where a second reactant B is injected uniformly along the reactor length instead of being co-fed at the inlet. This is the continuous-flow analogue of semibatch: C_A is high (fully fed at inlet), C_B is kept low (distributed addition). ODE with distributed feed term: dFA/dτ = rA(CA,CB)·FA₀, dFB/dτ = FB₀_dist − rB·FA₀, where FB₀_dist = total_FB₀ / tau_pfr is the distributed feed rate per unit space time. Concentrations at any τ: CA = FA/Q(τ), CB = FB/Q(τ), Q(τ) = Q₀ + (FB₀_dist/CB_feed)·τ (volumetric flow grows along length). Same RK4 solver as the existing PFR; just extend the state vector with FB and Q and add the distributed source. Implementation: a flag sideInjection: boolean and new param FB0_side (total B feed rate, mol/s) on the existing PFR node data. No new node type needed — extend pfr. PFRNode.tsx shows a "⊕B" badge when sideInjection === true. New function: sideFeedPFR() in src/math/unitModels.ts.
Enables
PFR = co-feed at inlet · CB high throughout → distributed B injection · CB low along length · higher S_R vs. co-feed PFR

15
Fixed-Bed Catalytic Reactor
Adds W_cat to node data · no SimulationParams migration needed (node-local)
A PFR operating on a catalyst weight basis rather than a volume basis. The Moles design equation: dFA/dW = −r′A where r′A is in mol/(kg_cat·s). This is the standard design equation from Fogler Ch. 10 for heterogeneous catalysis. Implementation: extend the existing PFR ODE path — when node.data.catalytic === true, swap the independent variable from τ (volume/Q) to W/FA₀. Node params: W_cat (kg catalyst), rho_bulk (kg/m³, default 1200), epsilon_bed (void fraction, default 0.4). Derived: V_bed = W_cat / (rho_bulk · (1 − epsilon_bed)). The Da equivalent is Da' = k'·W/FA₀ where k' is the rate constant in mol/(kg·s·(mol/L)^n). Display: node shows "W: X kg" instead of "τ: X s". Ergun pressure drop (already implemented for PFR) integrates naturally over the catalyst bed. New node type: fixedbed with a distinct icon (stacked-layer symbol). New function: catalyticPFR() in unitModels.ts.
Enables
volumetric-basis PFR only → weight-basis · catalyst design · W/FA₀ Da analogue · bed sizing

🔬 Graphify: store monoculture — no new direct useSimulatorStore calls in Phase 4
✗ Each new Phase 4 component (PropertiesPanel, ReactionBuilderModal, SemibatchNode) calls useSimulatorStore(s => s.X) directly, as all existing components do. This adds more edges to the god node, raising useSimulatorStore from 67 to 80+ cross-community edges by end of Phase 4.

The store itself does not change — only access patterns. Selector hooks also enable React.memo bailouts that direct subscriptions cannot, so Phase 4 gets a performance improvement for free.

Custom preset: factory function vs. eval() / dynamic code
✗ Build the rate law as a string expression and call eval() or new Function() at runtime. Tempting because it lets users type "k*Ca**2" directly — but eval is a CSP violation, impossible to typecheck, and a security footgun.
✓ Template factory: buildCustomPreset(cr: CustomReaction) takes the rate type enum and constructs a closure using the species array. Power Law template: (C, T, kP) => arrhenius(...) * reactants.reduce((acc, s) => acc * (C[s.label] ?? 0) ** s.stoich, 1). All code paths are compile-time safe, testable, and follow the existing arrhenius() pattern.

The three built-in templates (Power Law, MM, LH) cover 95% of undergraduate ChE kinetics. Adding a new template later is a one-function addition to the factory — not an architecture change.

Semibatch ODE: moles basis vs. concentration basis
✗ Write the semibatch ODE on a concentration basis: dCA/dt = rA + (CA_in − CA)·vB/V. This form looks familiar but has a dilution term that is easy to get wrong when V changes — especially the sign of the dilution term for products.
✓ Moles basis: state vector [NA, NB, NR, V]. Concentrations are derived: CA = NA/V, etc. The four ODEs are simple and unambiguous. RK4 steps are identical to the existing batch reactor ODE structure in unitModels.ts. Xa and S_R computed from final moles.

The moles-basis formulation is standard in Fogler Appendix B and leaves no room for sign errors on the dilution term. Always derive concentrations, never track them as state variables when V is changing.

Floating popover state: sessionSlice vs. localStorage
✗ Persist the popover open/closed state in localStorage like the accordion section state. Sounds convenient, but the popover is transient UI — if the user navigates away and back, restoring a floating panel that obscures the canvas is disorienting.
✓ Store popover open state in sessionSlice.paramPopoverOpen: boolean (default false). This slice is in-memory only — not persisted to localStorage. The popover always starts closed on page load. The section open states within the popover still persist to localStorage via the existing rsi-param-sections key.

Distinction: which sections are expanded (meaningful preference to persist) vs. whether the popover is open (transient session state). Get the persistence boundary right the first time — changing it later means a localStorage migration.

Selectivity analysis data: computed in solver vs. derived on render

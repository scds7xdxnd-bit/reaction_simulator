🔑 Architectural decisions to get right upfront
Feed overrides, not replacement
✗ Move Ca0/T_feed entirely off SimulationParams onto FeedNode.data
✓ Per-node FeedNode.data.Ca0? resolved as nodeData.Ca0 ?? params.Ca0 — global stays as default

Breaking change risks every saved file and example. The override pattern is non-breaking, unambiguous, and lets the user share a global default while customising individual feeds.

Sweep engine is pure — results live in the slice
✗ Put sweep logic inside useSweep hook with store reads inside the engine
✓ sweepEngine.ts is a pure function (data in, data out). useSweep dispatches config and reads sweepSlice.

Mirrors the networkSolver/useSimulation split that already works. Pure function is testable in one line. Hook is thin. No entanglement.

Context menu state goes in sessionSlice
✗ Create a new contextMenuSlice for the menu's visible/x/y/targetId
✓ Add {menuVisible, menuX, menuY, menuTargetId} to the existing sessionSlice

This is transient UI state — exactly what sessionSlice already holds (selectedNodeId, simulationMode). No need for a new slice for a floating menu that has no domain meaning.

Design System before Interaction features
✗ Build context menu now using ad-hoc inline styles, migrate later
✓ P2 (design tokens + ui/ components) lands first, then P3 uses the primitives

The context menu is the first net-new UI surface in Phase 2. If built before ui/Button exists it accumulates debt immediately. One week of design system pays off on every feature that follows.

The operating principle for Phase 2: Phase 1 cut seams — Phase 2 exploits them. solveNetwork is pure → sweep is a loop. ChemistryModel is injected → reversible reactions are a new preset, not a new solver. rk4Step is isolated → batch reactor reuses it directly. Every new feature should feel like plugging into an existing socket, not cutting a new one.
